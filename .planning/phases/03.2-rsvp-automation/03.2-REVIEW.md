# Phase 3.2: Code Review Report (RSVP & Status Automation - Revised)

**Reviewed:** 2026-05-16
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

The revised design for Phase 3.2 significantly improves security by moving RSVP processing to a server-side PocketBase hook and addresses previous concerns regarding status overrides and transparency. However, several **BLOCKER** issues remain in the logic and security architecture. Most notably, the "HMAC Secret" is still leaked to the frontend, and the status automation algorithm contains logic bugs that ignore actual attendance and make the "Inactive" status nearly impossible to escape automatically.

## Critical Issues

### CR-01: HMAC Secret Leak to Frontend (Security)

**File:** `.planning/phases/03.2-rsvp-automation/03.2-02-PLAN.md` (Task 1)
**Issue:** The plan requires the frontend `communicationService` to fetch the `HMAC_SECRET` from `appSettings` to sign RSVP tokens.
1. **Exposure:** Any user with the "admin" role can now see the master secret in their network tab or console. 
2. **Forgeability:** If the secret is leaked, anyone can forge RSVP links for any singer/event, bypassing the security intended by the signature.
**Fix:** 
RSVP links should be signed on the server. Implement a custom PocketBase route (e.g., `POST /api/sign-rsvp`) that takes a payload and returns a signature, or handle the token generation within a server-side "Send Email" hook. **Never** send cryptographic secrets to the browser.

### CR-02: "Sticky Inactive" Logic Bug (Logic)

**File:** `.planning/phase-3/SPEC.md` (Section: Automatic Status Logic)
**Issue:** The transition rules prioritize "Inactive" over "Active". If a singer is marked "Inactive" because they missed the 3 most recent past performances, they cannot automatically return to "Active" status by RSVPing "Yes" to future events.
- The logic checks the "3 most recent past performances". Even if a singer RSVPs "Yes" to the next 5 upcoming events, the "3 most recent past" will still be the missed ones until those future events actually occur and the singer attends them.
**Fix:** 
Change the priority or logic. An explicit "Yes" RSVP to an upcoming performance where `isOpenForRSVP` is true should likely override the "Inactive" state (transitioning the user to "Active (Future)" or "Active (Current)"), or the Inactive check should only apply if there are no upcoming "Yes" RSVPs.

### CR-03: Attendance Ignored in "Missed" Calculation (Logic)

**File:** `.planning/phase-3/SPEC.md` (Section: Automatic Status Logic)
**Issue:** The "Inactive" criteria only checks the `rsvp` status: "The RSVP status for all 3 is either 'No' or 'Pending'". 
- This ignores the `attendance` field. If a singer RSVPs "Yes" but is marked "Absent" by an admin, the automation will still treat them as "Active". A singer could theoretically stay "Active" indefinitely by RSVPing "Yes" and never showing up.
**Fix:** 
Update the logic to check both RSVP and Attendance. A performance is "missed" if `(rsvp == 'No' || rsvp == 'Pending')` OR `(attendance == 'Absent')`.

## Warnings

### WR-01: Insecure "Safe Default" for HMAC Secret

**File:** `.planning/phases/03.2-rsvp-automation/03.2-01-PLAN.md` (Task 2)
**Issue:** The plan suggests using a "safe default" if the `HMAC_SECRET` is missing from settings. 
**Risk:** There is no such thing as a "safe default" for an HMAC secret. Using a hardcoded or predictable default allows attackers to forge tokens on any system where the admin hasn't explicitly set a secret.
**Fix:** 
The hook should throw a 500 error and log a critical failure if the secret is missing. Do not fallback to a default.

### WR-02: Inconsistent Event Filtering (Quality)

**File:** `.planning/phase-3/SPEC.md`
**Issue:** The logic for past performances uses `created > profile.created`. 
- This filters based on when the *event record* was created in the database, not when the event actually occurs. If a performance was scheduled months in advance (before a singer joined), it won't be counted, even if it happens after they joined.
**Fix:** 
Use the event date for filtering: `event.date > profile.created`.

### WR-03: Race Conditions in Status Recalculation

**File:** `.planning/phases/03.2-rsvp-automation/03.2-01-PLAN.md` (Task 3)
**Issue:** The `onModelAfterUpdate` hook on `eventRosters` triggers a full profile recalculation. If an admin performs a bulk update (e.g., marking attendance for 50 people at once), 50 hooks will fire concurrently. 
**Risk:** While PocketBase is performant, multiple concurrent updates to the same `profiles` record could lead to "stale update" errors (400) or redundant database load.
**Fix:** 
Ensure the hook uses the existing transaction and considers debouncing or checking if the status actually *needs* to change before committing the update to the profile.
